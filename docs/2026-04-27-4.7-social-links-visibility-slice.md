# Checkpoint 4.7 Implementation Slice - Social Links and Profile Visibility Controls

## Objective

Extend the current member-profile foundation so social and professional links move from stored data to governed presentation behavior.

This slice should preserve the current privacy-first baseline:
- profile fields remain private by default,
- members explicitly choose visibility,
- public presentation requires review and approval,
- internal member views and public views use different rendering rules.

## Current Verified Baseline

- Member profile editing is live for core fields, LinkedIn, professional links, birthday preferences, and field-level visibility overrides.
- Members can submit fields for public review by switching visibility states to `submitted_for_public_review`.
- Admins can see pending profile review submissions and approve or reject them.
- Public pages do not yet render approved member profile data.

## Smallest Complete 4.7 Slice

Deliver one governed profile-presentation path for approved social/profile fields without widening scope into conference memory or full public storytelling.

### In Scope

1. Add a member-directory/profile rendering rule set that only exposes fields marked `public_approved`.
2. Normalize how LinkedIn and professional links are transformed into public-safe display items.
3. Make admin review decisions update the effective public-display state clearly and predictably.
4. Show approved public-profile content in one controlled presentation surface.
5. Keep member-only and admin views richer than the public view.

### Out Of Scope For This Slice

- conference reflection threads,
- honorary/memorial public pages,
- broader public storytelling workflows,
- auto-posting to social platforms,
- image moderation beyond the existing profile-photo controls.

## Implementation Targets

### API

Primary files:
- `apps/api/src/server.mjs`
- `apps/api/test/server.test.mjs`

Changes:
- Add a read model/helper that resolves the effective public-facing profile fields from `profile_visibility_json`.
- Ensure only fields with `public_approved` visibility are returned from any new public-profile endpoint.
- Tighten review decisions so `approved` and `rejected` outcomes are reflected consistently in stored visibility state and returned payloads.
- Return normalized link payloads for LinkedIn and professional links with safe labels and `https`-only URLs.

Suggested endpoint slice:
- `GET /api/public/members` or `GET /api/public/member-profiles`

Response shape should include only:
- member identity fields approved for public display,
- approved professional title/company/bio/expertise fields,
- approved LinkedIn/professional links,
- approved photo URL if allowed.

It should exclude:
- phone,
- birthday data,
- member-only/internal visibility states,
- raw visibility metadata,
- admin review notes.

### Web

Primary files:
- `apps/web/src/templates.mjs`
- `apps/web/test/server.test.mjs`
- `apps/web/public/styles.css` only if the existing layout cannot support the new public presentation cleanly.

Changes:
- Add a controlled public-rendering section for approved member highlights.
- Keep the presentation restrained and governance-aware rather than directory-heavy.
- Show links only when approved and present.
- Keep the member profile editor unchanged except for any needed clarity text about what `public_approved` means after review.
- Keep the admin review queue focused on approval state, requested fields, and decision actions.

## Test Slice

### API tests to add

In `apps/api/test/server.test.mjs`:

1. Approved public-profile fields are returned from the public-profile endpoint.
2. Non-approved fields remain absent even when profile data exists.
3. Rejected fields remain excluded from the public-profile endpoint.
4. Public links are normalized and reject non-`http`/`https` schemes.
5. Birthday, phone, and internal-only fields never leak through the public-profile response.

### Web tests to add

In `apps/web/test/server.test.mjs`:

1. Public route renders approved member-profile highlights when the API returns approved data.
2. Public route does not render unapproved private/member-only fields.
3. Approved LinkedIn/professional links render as anchors.
4. No admin/member-only review metadata appears in public HTML.

## Validation Steps

Use the repo validation order with the narrowest checks first:

1. `node --test --test-isolation=none apps/api/test/server.test.mjs --test-name-pattern "public profile|member profile"`
2. `node --test --test-isolation=none apps/web/test/server.test.mjs --test-name-pattern "public|profile"`
3. `npm run test`
4. `npm run build`

If the slice touches validation logic or templates broadly, finish with:

5. `npm run ci`

## Acceptance Criteria

- A member can submit social/profile fields for public review.
- An admin can approve or reject those fields.
- Only approved fields appear on the controlled public surface.
- Member-only/admin-only/private fields remain hidden publicly.
- Approved public links are rendered safely and consistently.
- Existing member/admin profile editing and review flows do not regress.
